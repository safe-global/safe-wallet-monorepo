# Web Project Structure

Use this before making changes under `apps/web/` or to web-only shared
files. Read the repo-wide conventions in
[../project-structure.md](../project-structure.md) first.

The detailed feature architecture reference is
[`apps/web/docs/feature-architecture.md`](../../../apps/web/docs/feature-architecture.md).

## Feature Folders

Web features live at `apps/web/src/features/<name>/`. Default shape:

```text
apps/web/src/features/<name>/
  feature.ts            # flat default export — components and services only, NEVER hooks
  index.ts              # createFeatureHandle + direct hook exports + public types
  components/
    <Component>/
      index.tsx         # always: components/<Component>/index.tsx
      <Component>.test.tsx
      <Component>.stories.tsx
  hooks/
    use<Hook>.ts        # exported directly from index.ts; not lazy
    use<Hook>.test.ts
  services/             # lazy-loaded heavy logic
  store/                # feature-local slices and selectors
  constants.ts          # shared symbols — do NOT export from hook files
  types.ts              # shared types — do NOT export from hook files
  enums.ts              # shared enums — do NOT export from hook files
```

Naming inside `feature.ts` decides stub behavior:

- `PascalCase` ⇒ component (renders `null` when not ready)
- `camelCase` ⇒ service (`undefined` when not ready — check `$isReady` first)
- Hooks belong in `index.ts`, never in `feature.ts`

Name a feature after the domain (`features/transactions`,
`features/assets`), not after a single component
(`features/pending-list`, `features/assets-list`). Sub-components live
as folders under `components/`, not as siblings of the feature folder.

Inside a feature, import siblings via relative paths (`./hooks/...`,
`./utils/...`, `./components/...`). Importing the feature's own
`index.ts` re-enters `createFeatureHandle` and produces a circular
dependency.
Feature-internal code should not import the feature handle or barrel
just to call a feature-local hook or flag. Import the direct hook,
constant, or shared feature-flag helper instead.

When a UI block has multiple children that must appear together, gate
it on both `$isReady` and `$isDisabled` from `useLoadFeature`. Gating
only on `$isDisabled` can render half-populated sections (e.g. an `OR`
divider above buttons that still render `null`).
Components returned by `useLoadFeature` can render `null` while
loading, so dividers, labels, or wrapper copy must not render alone
before the lazy children are ready.

`useLoadFeature` must be called at the top level of the consuming
component, not inside a nested wrapper component. Wrappers like
`const FeatureLoader = () => useLoadFeature(F)` violate Rules of
Hooks and break stub rendering.

When you swap out a wrapper component (e.g. `Header` → `Topbar`),
enumerate every behavior the old wrapper triggered (sidebar toggles,
batch-open state, notification renewal hooks) and confirm each is
handled or intentionally dropped with justification. A "drop-in
replacement" must read every caller-supplied prop or change its export
name so callers update intentionally.

The feature's barrel exports the feature handle, lightweight public
hooks, and public types only. Implementation components and services
stay behind `useLoadFeature` or local imports. Constants and IDs that
are not part of the lazy-loaded contract should be imported directly
from their component file rather than placed in `feature.ts`.

Copy:

- `apps/web/src/features/__core__/createFeatureHandle.ts` for the core
  handle/proxy primitives.
- Any feature whose `feature.ts` matches the flat-export pattern above.

Avoid copying:

- Any feature that adds `lazy()` calls inside `feature.ts` or nests its
  exports under `components` / `hooks` / `services` keys.
- Any feature that re-exports internal components or services through
  its barrel.

## UI And Styling

Theme tokens live in `packages/theme`. Web consumes them via
`var(--color-*)` in CSS modules and via the generated MUI theme. Always
use theme variables; do not introduce hex colors in JSX `className`
strings or inline `sx` overrides.

shadcn primitives sit under `apps/web/src/components/ui/*`. They are
regenerated by future shadcn imports — do not edit those files in
place. Customize via wrapper components, the Tailwind config, or
`cn()` overrides at call sites.

Long inline Tailwind class lists (multiple layout, scrollbar, and
pseudo-class utilities together) belong in a `*.module.css` class.
Keep only theme and high-level layout utilities inline.

When a component is loaded via `React.lazy()`, it suspends on first
render. Wrap such consumers in a `<Suspense>` boundary or use
`useLoadFeature` so the suspension is handled by the feature stub
mechanism. Plain `lazy()` without a `Suspense` ancestor will throw at
render in non-production paths.

## Bundle Hygiene

- Static logos and decorative SVGs use CSS `background-image`, CSS
  masks, or inline SVG. `next/image` adds measurable bundle weight
  without runtime benefit for fixed assets — observed regressions
  around 140 KB on the main bundle.
- Do not embed assets as base64 data URIs in source files; host on
  `apps.safe.global` and reference by URL.
- When combining Motion animations with Tailwind transform classes,
  split responsibilities across two elements (layout/centering on the
  parent, animation on the child). Motion's inline `transform`
  overrides Tailwind transform utilities.

## Testing

Unit tests import `render` and `renderHook` from
`apps/web/src/tests/test-utils.tsx`, never directly from
`@testing-library/react`. The wrapper injects providers (Redux store,
theme, router, feature handles) the components and hooks under test
depend on.

Pair the test description with the strongest assertion you can. A test
named "displays the total sum correctly" must assert the total, not
just `toBeInTheDocument()` on a digit fragment.

Use `data-testid` selectors and stable keys for assertions. Fall back
to text matching only when the test is specifically about rendering
text. When refactoring a component, preserve its existing
`data-testid` attributes; if a refactor renames a component, update
`apps/web/cypress/**` in the same PR.

Cypress preserves localStorage between tests in the same spec. Tests
that depend on storage state (address book, login, persisted settings)
must clear it explicitly in `beforeEach` (or per test) — otherwise
state from earlier tests pollutes later ones.

For Cypress feature flag toggling, use
`main.injectChainFeature({...})` from
`apps/web/cypress/e2e/pages/main.page.js`. The chains config payload is
paginated (`{ results: Chain[] }`), and feature flags live on each
`results[i].features`, so any custom interceptor must mutate
per-result features, not the top-level response. Avoid
`cy.contains(<tag>, ...)` and tag-based queries — select by
`data-testid`.

## Persistence And Data Portability

When you add a slice to `exportAppData`, add the matching parser in
`useGlobalImportJsonParser` / `ImportDialog` in the same PR.
Otherwise users export, re-import, and silently lose data.

When persisting RTK Query cache keys, the persisted form is JSON-
stringified — match the actual serialized form (with quotes for string
args) in any persistence filter.

## Next.js Specifics

`router.query.x` can be `string[]` in Next.js — a repeated query param
produces an array. Check `typeof x === 'string'` before casting; do
not force-cast.

Global header CTAs and Safe App links must build Safe-scoped routes
from loaded Safe state, not optional query params. If the target route
is blocked for undeployed Safes, gate the CTA on the same deployment
state instead of letting users click into a rejected flow.
